## 1. 产品概述

ChatDeepSeek CLI 是一个运行在终端（命令行）中的 Python 对话应用。它通过 DeepSeek 官方 API 实现与 AI 模型的连续对话，核心设计目标是：

- **极致省 Token**：精准控制每次发送给模型的上下文量，减少不必要的 Token 浪费
- **透明可控**：所有自动优化操作都会向用户通报，并可手动介入
- **多会话并行**：每个主题独立一个会话，上下文完全隔离，互不污染
- **本地化存储**：对话数据以可读的 JSONL 格式保存在本地，随时查看、备份

### 与其他工具的区别

| 特性 | ChatDeepSeek CLI | DeepSeek 网页版 | 其他 CLI 工具 |
|------|:---:|:---:|:---:|
| Token 优化（滑动窗口+语义召回） | ✅ | ❌ | ❌ |
| 多会话独立管理 | ✅ | ✅ | 部分 |
| `@` 锚点历史召回 | ✅ | ❌ | ❌ |
| 自动摘要压缩 | ✅ | ❌ | ❌ |
| 发送前 Token 预览 | ✅ | ❌ | ❌ |
| 终端流式 Markdown 渲染 | ✅ | ✅ | 部分 |
| 本地 JSONL 存储 | ✅ | ❌ | 部分 |

## 2. 安装与启动

### 2.1 环境要求

- **操作系统**：Windows / macOS / Linux
- **Python**：3.10 或更高版本
- **网络**：能访问 `api.deepseek.com`
- **DeepSeek API Key**：[在此获取](https://platform.deepseek.com/api_keys)（需要注册并充值）

### 2.2 安装步骤

```bash
# 第一步：进入项目目录
cd chat_deepseek

# 第二步：安装依赖（约 1~3 分钟，取决于网络速度）
pip install -r requirements.txt

# 第三步：配置 API Key
# 用任意文本编辑器打开 .env 文件
# 将 DEEPSEEK_API_KEY=your_api_key_here 中的 your_api_key_here
# 替换为你的真实 Key，例如：
#   DEEPSEEK_API_KEY=sk-abc123def456...
```

### 2.3 依赖包说明

| 包名 | 用途 | 是否必需 |
|------|------|:---:|
| `openai` | 调用 DeepSeek API（兼容 OpenAI 协议） | ✅ 必需 |
| `python-dotenv` | 读取 `.env` 配置文件 | ✅ 必需 |
| `rich` | 终端彩色美化 + Markdown 渲染 | ✅ 必需 |
| `numpy` | 语义搜索的向量计算 | ✅ 必需 |
| `tiktoken` | 精确 Token 数量估算 | ⚠️ 可选 |
| `sentence-transformers` | 语义相似度精确计算 | ⚠️ 可选 |

> **可选依赖说明**：不装也能跑。`tiktoken` 不可用时，Token 估算回退到字符数粗略计算（略有偏差）；`sentence-transformers` 不可用时，语义搜索回退到 TF-IDF + 关键词匹配（精度略降，不影响基本使用）。

### 2.4 启动

```bash
python main.py
```

启动后会显示欢迎面板，自动进入最近一次使用的会话。首次运行时自动创建名为「默认」的空会话。

### 2.5 退出

- 输入 `!exit` 优雅退出
- 或按 `Ctrl + C`
- 退出时所有对话自动保存

```
[会话名] [模型] > _
```

| 元素 | 颜色 | 含义 |

### 3.2 颜色约定

| 颜色 | 用途 |
|------|------|

| 灰/暗色 | 系统优化通报、辅助信息 |


这表明系统自动完成了上下文筛选操作。

## 4. 对话基础

### 4.1 发起对话

在提示符后直接输入内容并按回车即可：

```
[编程] > 请用 Python 写一个快速排序
```

AI 的回复会以**流式打字**效果逐字显示（SSE 流式传输），支持 Markdown 代码高亮。

### 4.2 多行输入

直接在终端输入时，按回车即发送。如需多行输入，有以下方式：

- **Windows PowerShell**：在行末输入 `` ` `` 后按回车继续输入（反引号续行）
- **粘贴**：直接粘贴多行文本，终端会自动处理换行

### 4.3 对话流程

每轮对话的内部流程如下：

```
用户输入
    ↓
解析 @ 锚点，提取召回关键词
    ↓
滑动窗口 + 语义召回 → 筛选历史消息
    ↓
组装系统提示（核心角色 + 知识摘要 + 临时角色）
    ↓
检查是否需要历史摘要压缩
    ↓
Token 溢出预防（超限时自动裁切）
    ↓
[如果开启了发送前确认 → 展示预览，等待确认]
    ↓
发送到 DeepSeek API（流式 SSE）
    ↓
接收回复（逐字打印）
    ↓
保存对话历史
    ↓
显示优化通报
```

## 5. 会话管理

会话即一个独立的对话上下文。每个会话拥有自己的历史记录、系统提示、模型选择和配置。

### 5.1 创建会话

```
/new <会话名> [模型]
```

- **会话名**：支持中文、英文、数字、下划线和短横线
- **模型**：可选参数，不指定则使用 `.env` 中的 `DEFAULT_MODEL`

```bash
/new 编程 deepseek-v4-flash     # 创建并指定模型
/new 写作                        # 创建，使用默认模型
/new 脑暴 deepseek-reasoner     # 创建推理会话
```

### 5.2 切换会话

```
/switch <会话名>
```

**前缀模糊匹配**：不必输入完整名称，输入前面几个字即可：

```bash
/switch 编          # 如果只有一个 "编程"，直接切换
/switch de          # 如果有 "demo" 和 "default"，列出候选
```

切换后会显示目标会话的最后一条消息摘要（前 60 字符）和相对时间：

```
已切换到 [写作]（最后消息: "请帮我拟定文章大纲……"，3小时前）
```

### 5.3 列出会话

```
/list
```

输出示例：

```
  * 编程 (deepseek-v4-flash)     12条消息    最后活动: 2分钟前
    大脑暴 (deepseek-reasoner)    8条消息    最后活动: 3小时前
    写作 (deepseek-v4-flash)      5条消息    最后活动: 1天前
```

- `*` 标记当前会话
- 按最后活动时间降序排列

### 5.4 重命名会话

```
/rename <新名称>
```

只影响当前会话，对话历史全部保留。磁盘上的对应文件同步重命名。

### 5.5 删除会话

```
/delete <会话名>
```

**需要输入 `y` 确认**，操作不可撤销。对话文件和元数据同时删除。

如果删除的是当前会话，系统会自动切换到其他会话；如果这是最后一个会话，会自动创建一个新的「默认」会话。

### 5.6 克隆会话

```
/clone <源会话> <新会话>
```

复制源会话的**全部内容**到新会话，包括：

- 所有历史消息（含时间戳）
- 系统提示设定（核心角色、知识库、临时角色）
- 模型选择和摘要设置

适用于需要在不破坏原对话的情况下尝试新思路，或交叉参考。

### 5.7 会话启动行为

- 启动时自动扫描 `chats/` 目录，重建所有会话
- 自动进入**最近活动**的会话
- 若目录为空，自动创建「默认」会话
- 遇到损坏的 JSONL 行自动跳过并提示

## 6. 系统提示（角色设定）

ChatDeepSeek 采用**三段式系统提示**，每轮对话时按优先级合并注入，而非全部打包发送。

### 6.1 三段式结构

```
┌──────────────────────────────────────────┐
│  ① 核心角色（每轮必带，≤3 句最佳）        │
├──────────────────────────────────────────┤
│  ② 知识库（首次完整版，后续自动摘要）      │
├──────────────────────────────────────────┤
│  ③ 临时角色（可持久或单轮）               │
├──────────────────────────────────────────┤
│  ④ 历史对话纪要（自动生成，若开启摘要）    │
└──────────────────────────────────────────┘
```

最终发送给 API 的 system 消息由上述内容用 `\n\n` 拼接而成。

### 6.2 核心角色 — `/core`

设定 AI 的基础人设和行为准则，每轮对话必定携带。

```
/core 你是一位资深的 Python 后端工程师，擅长异步编程、数据库优化和代码审查。回答简洁专业。
```

- 建议控制在 3 句以内，过长的核心角色会每轮都消耗 Token
- 设置后自动保存，重启不丢失
- 查看：`/role show`（会同时显示核心角色和临时角色）

### 6.3 知识库 — `/doc`

注入大段参考资料（如文档片段、技术规范、项目需求等）：

```
/doc 项目描述：这是一个电商后端系统，使用 FastAPI + PostgreSQL，需要支持高并发秒杀场景。技术栈包括 Redis 缓存、RabbitMQ 消息队列、Docker 部署...
```

**摘要机制（自动省 Token）：**

1. 第一次对话：系统将完整文档注入上下文
2. 首次回复后，自动询问是否调用模型生成摘要（约消耗 200 tokens）
3. 生成 2~3 句关键要点摘要，缓存在会话元数据中
4. 之后每轮对话：只发送摘要，不再发送完整文档

**手动召回完整文档：**

```
/recall_doc
```

将完整文档重新注入**仅本轮**上下文。不影响已有的角色设定（`/role` 不受影响）。

### 6.4 临时角色 — `/role`

追加临时的行为指令，可以是单轮也可以是持久：

```bash
/role 请用简体中文回答，代码注释用英文     # 持久生效
/role 本次回答请特别关注安全性方面         # 持久生效
/role off                                  # 关闭临时角色
/role show                                 # 查看当前所有角色设定
```

**行为规则：**

| 操作 | 行为 |
|------|------|
| `/role <描述>` | 设定持久角色，每轮生效，直到 `/role off` |
| `/role off` | 清除临时角色，立即生效并保存 |
| `/role show` | 显示当前的核心角色和临时角色 |

> **注意**：`/role` 不影响 `/core` 设定的核心角色，两者共存。

## 7. 上下文控制（省 Token 核心）

这是 ChatDeepSeek 最重要的功能。每次发送消息给 API 前，系统会自动筛选哪些历史消息真正需要带上。

### 7.1 三层筛选策略

```
全部历史消息
    │
    ├── 最近 N 轮（静态窗口）──────── → 全部带入 ✅
    │
    ├── 窗口外的旧消息
    │   ├── 语义高分（相似度 ≥ 阈值）→ 召回带入 ✅
    │   └── 语义低分                  → 丢弃 ❌
    │
    └── 更早的远历史
        ├── 压缩比 ≥ 5:1              → 压缩为纪要 ✅
        └── 压缩比 < 5:1              → 丢弃 ❌
```

### 7.2 静态滑动窗口

- 默认保留最近 **10 轮**（即最近 20 条消息，每轮 = 1 用户 + 1 助手）
- 窗口大小由 `.env` 中的 `MAX_WINDOW` 控制
- 窗口内的消息**全部保留**，不做任何筛选

### 7.3 语义召回

对于落在窗口外的旧消息，根据与当前提问的**语义相似度**决定是否带入：

1. 使用本地嵌入模型（`sentence-transformers`）或 TF-IDF 计算相似度
2. 仅保留相似度 ≥ `SIMILARITY_THRESHOLD`（默认 0.5）的消息
3. 最多带回 `SEMANTIC_TOP_K`（默认 4）条
4. 全部本地计算，**不消耗 API Token**
5. 若 `sentence-transformers` 不可用，自动回退到 TF-IDF 匹配

### 7.4 自动摘要压缩 — `/summarize`

将非常旧的历史消息压缩为一段简要纪要，大幅节省 Token。

**开启/关闭：**

```bash
/summarize on       # 开启
/summarize off      # 关闭
```

**触发条件（两个条件同时满足）：**

1. 早于窗口的旧消息累计 Token 数 ≥ `COMPRESSION_THRESHOLD_TOKENS`（默认 2000）
2. 预估压缩比 ≥ `COMPRESSION_MIN_RATIO`（默认 5.0，即压缩后 ≤ 原长的 20%）

**执行过程：**

- 将符合条件的旧消息打包，静默调用模型生成"对话纪要"
- 纪要内容标注时间范围，替换掉原始消息
- 压缩生成的 Token 消耗会计入总体统计
- 原始消息不会丢失，仍可通过 `@` 锚点临时召回

### 7.5 Token 溢出预防

若最终组装的上下文仍超过模型限制（约 64K tokens），系统会自动裁切：

1. **第一轮**：从最旧的 `assistant` 消息开始丢弃（保留 system 和 user 消息）
2. **第二轮**：如果还是超限，丢弃非最后一条的 `user` 消息
3. 裁切数量会显示在优化通报中

### 7.6 优化通报解读

```
[优化] 滑动窗口 10 轮 | 语义召回 2 条 | 远历史已压缩（节省 1200 tokens）
```

| 信息 | 含义 |
|------|------|
| 滑动窗口 N 轮 | 最近 N 轮对话被完整保留 |
| 语义召回 M 条 | 从旧消息中召回了 M 条相关内容 |
| 关键词召回 M 条 | 通过精确关键词匹配召回了 M 条 |
| 远历史已压缩 | 旧消息被压缩为纪要，节省了 X tokens |
| 裁切 N 条超限消息 | Token 仍然超限，裁切了 N 条旧消息 |

## 8. 历史召回（@ 锚点）

在提问中使用 `@` 前缀，可以主动召回历史消息注入当前上下文。

### 8.1 语义召回：`@关键词`

```
@网络超时 结合之前那个问题，给出重试策略
```

系统会在窗口外的旧消息中搜索与"网络超时"语义相关的内容，相似度最高的几条自动注入本轮上下文。

- 支持多个关键词：`@网络超时 @重试 @TCP 汇总分析`
- 仅从**窗口外**的旧消息中搜索（窗口内的已经全部带入了）
- 每个关键词召回 1~2 条最相关的消息
- 与窗口内消息自动去重

### 8.2 精确匹配：`@"精确短语"`

```
@"TCP连接超时" 和 @"HTTP请求重试" 有什么区别
```

使用双引号触发**关键词精确匹配**（不依赖语义模型），适用于记得历史中出现了某个确切短语时使用。

### 8.3 未命中提示

如果 `@` 锚点未能匹配到任何历史消息：

```
⚠ 未找到与 "网络超时" 相关的历史消息
```

### 8.4 @ 锚点解析规则

| 输入 | 类型 | 匹配方式 |
|------|------|------|
| `@网络超时` | 语义 | 语义相似度 |
| `@"网络超时"` | 精确 | 关键词计数匹配 |
| `参考 @超时 @重试 内容` | 多个语义 | 合并检索 |
| `@网络超时 @"TCP重试"` | 混合 | 分别检索后合并去重 |

### 8.5 技术细节

- `@` 锚点**不会**被发送给 API，系统会剥离它们，只发送清理后的纯文本
- 历史上保存的是**原始输入**（含 @ 锚点），便于回顾
- 语义召回完全在本地计算，**不消耗额外 API Token**

## 9. Token 统计与费用

### 9.1 统计命令

```bash
/tokens last         # 上一轮对话的 Token 消耗
/tokens session      # 当前会话的累计 Token 消耗
/tokens total        # 本次运行所有会话的累计 Token 消耗
```

### 9.2 输出示例

```
上一轮消耗
  Prompt tokens              1234
  Completion tokens           567
  Total tokens               1801
  费用（≈ 参考价）           $0.000256（以 API 账单为准）
```

### 9.3 费用计算

- 使用内置价格表计算，仅作参考
- **最终费用以 DeepSeek 官方账单为准**

### 9.4 统计范围

| 统计项 | `/tokens session` | `/tokens total` |
|--------|:---:|:---:|
| 正常对话 | ✅ | ✅ |
| 摘要压缩生成 | ❌ | ✅ |
| 文档摘要生成 | ❌ | ✅ |
| 所有历史会话 | ❌ | ✅ |

> **说明**：`/tokens session` 只统计当前会话的对话 Token，不包含后台摘要操作。`/tokens total` 包含所有 Token 消耗。

## 10. 发送前确认

### 10.1 开关

```bash
/confirm on        # 开启：每次发送前预览
/confirm off       # 关闭（默认）
```

### 10.2 预览内容

开启后，每次发送前会显示：

```
═══ 发送前确认（约 1234 tokens）═══
```

## 12. 数据存储与持久化

### 12.1 存储结构

```
chat_deepseek/
└── chats/
    ├── .meta/                   # 会话元数据目录
    │   ├── 编程大师.json         # 「编程大师」会话的元数据
    │   ├── 新朋友.json           # 「新朋友」会话的元数据
    │   └── ...
    └── *.jsonl                  # 对话数据文件
        ├── 编程大师.jsonl        # 「编程大师」的对话历史
        ├── 新朋友.jsonl          # 「新朋友」的对话历史
        └── ...
```

### 12.2 元数据格式（`.meta/会话名.json`）

```json
{
  "name": "编程大师",
  "model": "deepseek-v4-flash",
  "created_at": "2026-06-20T10:00:00+00:00",
  "core_role": "你是一位资深的 Python 后端工程师...",
  "knowledge_summary": "该文档描述了电商系统的技术架构...",
  "knowledge_doc": "项目描述：这是一个电商后端...（完整原文）",
  "temp_role": "请用简体中文回答",
  "temp_role_persistent": true,
  "summary_enabled": false,
  "history_summary": "",
  "summary_start_idx": 0
}
```

### 12.3 对话格式（`.jsonl`）

每行一条 JSON 消息，每条一行，人类可读：

```json
{"role": "user", "content": "你好", "timestamp": "2026-06-20T10:00:00+00:00"}
{"role": "assistant", "content": "你好！有什么可以帮你的？", "timestamp": "2026-06-20T10:00:05+00:00", "token_usage": {"prompt": 15, "completion": 12}}
```

### 12.4 数据安全

- **原子写入**：先写 `.tmp` 临时文件，写入完成后再替换正式文件，防止系统崩溃导致文件损坏
- **自动修复**：启动时扫描 JSONL 文件，自动跳过损坏的行并提示用户
- **透明可读**：所有存储格式为纯文本 JSON/JSONL，可用任意文本编辑器查看和编辑
- **手动备份**：直接复制整个 `chats/` 目录即可完成备份

## 13. 配置参数完全参考

编辑 `.env` 文件调整程序行为。以下为所有支持参数：

### API 连接

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `DEEPSEEK_API_KEY` | (必填) | DeepSeek API 密钥，从 platform.deepseek.com 获取 |
| `DEEPSEEK_BASE_URL` | `https://api.deepseek.com` | API 地址，一般不需要修改 |

### 窗口与上下文

| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| `MAX_WINDOW` | `10` | 1~50 | 滑动窗口保留的对话轮数。越大模型能"记住"越多，但越费 Token |
| `DEFAULT_MODEL` | `deepseek-chat` | 任意模型ID | 新建会话时默认使用的模型 |
| `SIMILARITY_THRESHOLD` | `0.5` | 0.0~1.0 | 语义召回相似度阈值。越大越严格（召回更少但更相关） |
| `SEMANTIC_TOP_K` | `4` | 1~20 | 语义召回最多带回的消息条数 |
| `MAX_OUTPUT_TOKENS` | `4096` | 1~65536 | 每次 AI 回复的最大 Token 数。越大回复越完整，也越贵 |

### 摘要压缩

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `COMPRESSION_THRESHOLD_TOKENS` | `2000` | 旧消息累计超过此 Token 数才开始考虑压缩 |
| `COMPRESSION_MIN_RATIO` | `5.0` | 最小压缩比（原长度/压缩后长度），低于此比例不压缩 |

### 重试策略

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `MAX_RETRIES` | `3` | API 调用失败时的最大重试次数 |
| `RETRY_DELAY` | `2` | 重试基础间隔（秒），实际间隔为 `delay × 2^attempt`（指数退避） |

### 参数调优建议

| 场景 | 建议调整 |
|------|----------|
| 想省钱 | `MAX_WINDOW=5`，`SEMANTIC_TOP_K=2`，`MAX_OUTPUT_TOKENS=2048` |
| 要更精准的回复 | `MAX_WINDOW=15`，`SIMILARITY_THRESHOLD=0.6`，`MAX_OUTPUT_TOKENS=8192` |
| 讨论代码/长文档 | `MAX_WINDOW=8`，`MAX_OUTPUT_TOKENS=16384`，配合 `/summarize on` |
| 闲聊/轻量对话 | `MAX_WINDOW=5`，`SIMILARITY_THRESHOLD=0.3`，`MAX_OUTPUT_TOKENS=2048` |
| 深度分析/头脑风暴 | `MAX_WINDOW=12`，使用 `deepseek-reasoner` 模型 |

## 14. 故障排除

### 14.1 启动失败

**现象**：提示"请先在 .env 文件中设置 DEEPSEEK_API_KEY"

**解决**：打开 `.env` 文件，确认 `DEEPSEEK_API_KEY=` 后面有正确的密钥，且没有多余的空格或引号。

### 14.2 认证错误（401/403）

```
✗ 认证失败，请检查 DEEPSEEK_API_KEY 是否正确
```

**原因**：API Key 无效或已过期。

**解决**：
1. 登录 [platform.deepseek.com](https://platform.deepseek.com)
2. 检查 Key 是否有效，是否需要创建新 Key
3. 确认账户余额充足

### 14.3 请求频率限制（429）

```
⚠ 请求频率限制，4 秒后重试...
```

**原因**：短时间内发送了太多请求。

**解决**：
- 系统会自动等待并重试（指数退避），一般不需要手动干预
- 若频繁出现，降低提问频率
- 检查是否有其他程序同时使用同一个 API Key

### 14.4 网络异常

```
⚠ 网络异常，4 秒后重试（2/3）...
```

系统会自动重试最多 3 次。如果 3 次全部失败，会显示最终错误信息。

### 14.5 数据文件损坏

```
⚠ 警告: chats/编程大师.jsonl 第 5 行损坏，已跳过
```

系统会自动跳过损坏行，继续加载其余数据。损坏行之后的内容不受影响。

### 14.6 语义搜索不工作

如果 `@关键词` 总是提示"未找到相关历史消息"：
- 尝试降低 `SIMILARITY_THRESHOLD` 到 `0.3`
- 尝试使用精确匹配 `@"关键词"` 代替语义匹配
- 检查 `sentence-transformers` 是否正常安装

### 14.7 Python 版本不兼容

**最低要求**：Python 3.10。如果使用更低版本：
- Windows：[python.org](https://python.org) 下载最新版
- macOS：`brew install python@3.12`
- Linux：`sudo apt install python3.12`（Ubuntu/Debian）

## 17. 技术架构

### 17.1 模块依赖图

```
main.py (入口)
  ├── config.py     ← .env 配置文件
  ├── ui.py         ← rich 终端库
  ├── session_manager.py
  │     └── chats/*.jsonl + .meta/*.json
  ├── context_builder.py
  │     └── semantic_search.py
  │           ├── sentence-transformers (可选)
  │           └── sklearn (TF-IDF 回退)
  └── openai 库 → DeepSeek API
```

### 17.2 上下文组装管线

```
1. parse_at_anchors()        → 解析 @ 锚点，分离召回词和纯文本
2. 滑动窗口计算               → 确定窗口内消息索引
3. semantic_search()         → 对窗口外消息做语义相似度计算
4. keyword_search()          → 精确关键词匹配
5. 去重合并                   → 窗口 > 语义 > 关键词
6. _build_system_prompt()    → 按优先级组装 system 消息
7. _prevent_overflow()       → Token 溢出裁切
8. 返回最终 messages 列表
```

### 17.3 API 调用流程

```
_handle_chat()
  → _build_context_with_compression()   # 构建上下文(含压缩判断)
    → ContextBuilder.build()            # 核心组装管线
    → _send_with_retry(silent=True)     # 静默摘要生成(如触发)
  → _send_with_retry()                  # 主对话调用
    → _call_api()                       # 流式 SSE
      → openai.chat.completions.create()
```

## 18. 常见问题

### Q: 我输入的 @关键词 会发给 API 吗？

不会。系统会在发送前剥离所有 `@` 锚点，API 只收到清理后的纯文本。但原始输入（含 @ 锚点）会保存在本地历史中。

### Q: 会话文件能删除吗？

可以。直接删除 `chats/` 下对应的 `.jsonl` 和 `.meta/` 下对应的 `.json` 文件。下次启动时程序会自动跳过不存在的文件。

### Q: 能在多台电脑间同步会话吗？

可以。将整个 `chats/` 目录复制到另一台电脑的相同位置即可。注意 `.env` 中的 API Key 也需要在新电脑上配置。

### Q: 支持代理吗？

支持。在 `.env` 中设置 `DEEPSEEK_BASE_URL` 指向代理地址，或设置系统环境变量 `HTTP_PROXY` / `HTTPS_PROXY`。

### Q: token 用太多怎么办？

1. 减小 `MAX_WINDOW`（如改为 5）
2. 开启 `/summarize on`
3. 减小 `MAX_OUTPUT_TOKENS`
4. 使用 `/confirm on` 在发送前查看预估消耗

### Q: 模型回复太长被截断怎么办？

增大 `MAX_OUTPUT_TOKENS`（如改为 8192 或 16384）。

### Q: 如何彻底重置所有数据？

删除 `chats/` 整个目录，下次启动时程序会自动重建空目录结构。
